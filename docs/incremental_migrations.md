# Incremental Migration Workflow

This guide describes the complete workflow for managing incremental database migrations in the VGNC ORM project.

## Overview

The incremental migration system provides a safe, tested workflow for making database schema changes over time. It includes automated testing, validation, and deployment safety checks.

## Migration Workflow Script

The project includes a comprehensive migration workflow script at `scripts/migration_workflow.py` that provides all necessary tools for migration management.

### Available Commands

```bash
# Show current migration status
python scripts/migration_workflow.py status

# Create a new migration
python scripts/migration_workflow.py create "add_user_email_field"

# Test migration on clean database
python scripts/migration_workflow.py test

# Test specific migration revision
python scripts/migration_workflow.py test --revision abc123def456

# Test rollback functionality
python scripts/migration_workflow.py test-rollback --steps 1

# Validate migration safety
python scripts/migration_workflow.py validate

# Validate specific migration file
python scripts/migration_workflow.py validate --file alembic/versions/abc123def456_add_field.py

# Show deployment checklist
python scripts/migration_workflow.py checklist
```

## Standard Development Workflow

### 1. Make Model Changes

First, modify your SQLAlchemy models:

```python
# src/vgnc_internal_orm/models/species.py
class Species(BaseCustomModel):
    # ... existing fields ...

    # Add new field
    common_name: Mapped[Optional[str]] = mapped_column(
        String(128),
        nullable=True,
        comment='Common name for the species'
    )
```

### 2. Generate Migration

```bash
python scripts/migration_workflow.py create "add_common_name_to_species"
```

This will:

- Generate a new migration file in `alembic/versions/`
- Run ruff to format the code automatically
- Show the generated migration details

### 3. Review Generated Migration

Check the generated migration file to ensure it contains the expected changes:

```python
# alembic/versions/2024_01_15_1430-abc123def456_add_common_name_to_species.py
def upgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.add_column('species', sa.Column('common_name', sa.String(length=128), nullable=True, comment='Common name for the species'))
    # ### end Alembic commands ###

def downgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.drop_column('species', 'common_name')
    # ### end Alembic commands ###
```

### 4. Test Migration

```bash
python scripts/migration_workflow.py test
```

This will:

- Create a temporary database
- Apply all migrations including your new one
- Test basic database operations
- Verify schema integrity

### 5. Test Rollback

```bash
python scripts/migration_workflow.py test-rollback --steps 1
```

This ensures your migration can be safely rolled back if needed.

### 6. Validate Safety

```bash
python scripts/migration_workflow.py validate
```

This checks for:

- Dangerous operations (DROP TABLE, DROP COLUMN, etc.)
- Migration formatting issues
- Consistency with current schema

### 7. Run Full Test Suite

```bash
pytest tests/test_incremental_migrations.py -v
pytest tests/test_alembic_baseline.py -v
```

## Migration Types

### Additive Migrations (Safe)

Adding new columns, tables, or indexes:

```python
def upgrade() -> None:
    op.add_column('species', sa.Column('new_field', sa.String(100), nullable=True))
    op.create_index('idx_species_new_field', 'species', ['new_field'])

def downgrade() -> None:
    op.drop_index('idx_species_new_field', table_name='species')
    op.drop_column('species', 'new_field')
```

### Modifying Existing Columns

When modifying existing columns:

```python
def upgrade() -> None:
    # Make nullable change first (safer)
    op.alter_column('species', 'display_name',
                   existing_type=sa.VARCHAR(length=100),
                   type_=sa.VARCHAR(length=200),
                   nullable=False)

def downgrade() -> None:
    op.alter_column('species', 'display_name',
                   existing_type=sa.VARCHAR(length=200),
                   type_=sa.VARCHAR(length=100),
                   nullable=False)
```

### Data Migrations

For migrations requiring data changes:

```python
def upgrade() -> None:
    # Add new column first
    op.add_column('species', sa.Column('computed_field', sa.String(100), nullable=True))

    # Populate data (use raw SQL for complex operations)
    connection = op.get_bind()
    connection.execute(
        text("UPDATE species SET computed_field = UPPER(SUBSTRING(display_name, 1, 3))")
    )

    # Make column non-nullable if needed
    op.alter_column('species', 'computed_field', nullable=False)

def downgrade() -> None:
    op.drop_column('species', 'computed_field')
```

## Safety Checks and Validation

### Automated Safety Checks

The workflow automatically detects:

- **Destructive Operations**: DROP TABLE, DROP COLUMN, TRUNCATE
- **Data Loss Risks**: Direct DELETE statements
- **Missing Indexes**: Foreign keys without indexes
- **Migration Formatting**: Proper upgrade/downgrade functions

### Manual Validation Checklist

Before deploying to production:

```bash
python scripts/migration_workflow.py checklist
```

✅ Backup database before proceeding
✅ Run migrations on staging environment first
✅ Test rollback procedures
✅ Validate migration doesn't contain destructive operations
✅ Ensure sufficient maintenance window

✅ Monitor performance after deployment

## Common Migration Patterns

### Adding a New Table

```python
def upgrade() -> None:
    op.create_table('new_table',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('name', sa.String(100), nullable=False),
        sa.Column('created_at', sa.DateTime(), server_default=sa.text('now()'), nullable=False),
        sa.PrimaryKeyConstraint('id')
    )
    op.create_index('idx_new_table_name', 'new_table', ['name'])

def downgrade() -> None:
    op.drop_table('new_table')
```

### Adding Foreign Key

```python
def upgrade() -> None:
    # Add column first (nullable)
    op.add_column('child_table', sa.Column('parent_id', sa.Integer(), nullable=True))

    # Add foreign key constraint
    op.create_foreign_key('fk_child_parent', 'child_table', 'parent_table',
                          ['parent_id'], ['id'])

    # Populate data if needed
    # Make non-nullable if required

def downgrade() -> None:
    op.drop_constraint('fk_child_parent', 'child_table', type_='foreignkey')
    op.drop_column('child_table', 'parent_id')
```

### Changing Column Type

```python
def upgrade() -> None:
    # For safe type changes (e.g., increasing VARCHAR length)
    op.alter_column('table_name', 'column_name',
                   existing_type=sa.VARCHAR(length=50),
                   type_=sa.VARCHAR(length=100))

def downgrade() -> None:
    op.alter_column('table_name', 'column_name',
                   existing_type=sa.VARCHAR(length=100),
                   type_=sa.VARCHAR(length=50))
```

## Troubleshooting

### Common Issues

#### 1. Migration History Mismatch

```bash
   # Check current state
   python scripts/migration_workflow.py status

   # Force stamp to specific revision (use carefully)
   alembic stamp <revision_id>
```

#### 2. Migration Fails on Test Database

- Check model imports in `alembic/env.py`
- Verify all models are properly imported
- Check for circular import issues

#### 3. Rollback Issues

- Ensure downgrade function is properly implemented
- Test rollback on staging first
- Consider data implications of rollback

### Getting Help

```bash
# Show migration history
alembic history

# Show current revision
alembic current

# Show head revision
alembic heads

# Show detailed revision info
alembic show <revision_id>
```

## Best Practices

### Development

1. **Small, Focused Migrations**: One logical change per migration
2. **Test Both Directions**: Always test upgrade and downgrade
3. **Use Descriptive Messages**: Clear migration descriptions
4. **Review Generated Code**: Check autogenerated migrations for accuracy

### Production Deployment

1. **Staging First**: Always test in staging environment
2. **Backup Strategy**: Create database backup before migration
3. **Rollback Plan**: Know how to rollback if needed
4. **Monitoring**: Watch for performance issues after deployment
5. **Communication**: Notify users of maintenance windows

### Code Quality

1. **Follow Naming Conventions**: Use descriptive migration names
2. **Add Comments**: Explain complex data migrations
3. **Handle Edge Cases**: Consider NULL values, empty tables
4. **Test Data Scenarios**: Test with realistic data volumes

## Version Control

- **Never commit migrations with data**
- **Keep migrations in chronological order**
- **Don't modify committed migrations** (create new ones instead)
- **Document breaking changes in migration descriptions**

This comprehensive workflow ensures safe, reliable database schema changes for the VGNC ORM project.
